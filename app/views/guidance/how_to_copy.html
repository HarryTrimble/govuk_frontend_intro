Design pattern
Check before you start:
how to reuse

Help users understand whether they should use your service before they start, so that they don’t waste time and effort

If you do this well, you’ll improve user satisfaction and reduce unnecessary calls and applications to your service team.


Last updated: 29 June 2016

  1. Read and mark the regulations
  2. List key verbs and nouns
  3. Analyse past applications

1. Read and mark the regulations

Users have to read tens of pages of guidance just to learn if they: need to fill out a form; which form(s) to fill out; what information to put in these forms. Users should not have to learn how a service works. It should be as obvious as possible.

Michael, Maria and I went through all the regulations, rules, legislation.

All of them:

‘exclusions’, ‘exemptions’, ‘permits’, ‘bespoke permits’. The user doesn’t see these are separate, just the rules. Users try to follow rules, as they don’t want to get in trouble with government.

We separated ‘decisions rules’ from ‘conduct rules’:

Decision rules say what type of permission a user needs. Conduct rules are what a user can and cannot do once they have permission. These are typically called conditions.

We separated decision rules into two types:

Things users can answer e.g. are you building a bridge?
Things data can answer e.g. are you working on a main river?

This meant will mark three types of information in the regulation. Each getting a colour.

  • Blue = decision rules answered by the user
  • Purple = decision rule answered by data
  • Brown = conduction rules or ‘conditions’

5. Write questions

Having groups and some understanding of language users use, meant Michael could attempt writing routing questions. These questions are far from perfect, but they’re to test our concept with users.

Example question:

'Are you building a new structure on, in, next to, or across a river?'

This questions includes 11 'exclusions', 'exemptions' and 'permits', which cover by a single questions, reducing the number of questions a user having to answer overall.

6. Make the phone script first s

The first version of the routing questions was a phone script. Maria wrote it and Michael iterated the wording of each question. This allowed us to quickly test our logic. Only after Rob has tested the script with real users over the phone, did I make the digital version of routing questions.

7. Phone testing

Rob tested the phone-script with a construction consultant and an environmental manager at Gatwick Airport.

These are the key take outs:

Both people sought further clarity on certain questions.

“How do I find out whether a river is a salmonid river?” “How do I know if my work will have an adverse effect on species/habitats?” “Where is the height measured from - the river base, from the river bed? Is it the lowest point of the river bed or the adjacent point of the river bed?”

Rationale behind certain stipulations is unclear..

“The works are not carried out in or within 100 metres - That one is going to be a challenge in its own right. One, how are you going to police that? Where does that number come from? If there was something to back that up i could understand it. 50 metres I could understand but 100 metres seems excessive. I would expect more than 10 metres and less than 50. You could be working on the River Thames, Severn, Ouze it does take away the opportunity to go in there and carry out works under an exemption. You’re going to be tangling yourself up with more twine.”

8. Not one big decision tree, lots of little ones

Turning legislation straight into software doesn’t have a good history.

To account for all the situations or activity users do, the logic of law is literally made into a massive, single decision tree. This results in hundreds of ‘hard coded’ links and pages. These bespoke bits of software are not reusable. Also as soon as legislation changes, they become worthless and have to be thrown away.

To try avoid this, we decided to make lots of little decision trees. One for bridges, one for scaffolding, one for cables and so on. Each is separate. This means when legislation on bridges changes for example, we only need to change the questions about bridges.

Each sets of questions carries a value (exclusion, exemption, permit) depending how a user answers. The user goes through each set of questions (from most common to less common) . We limit the questions users have to answer, by having a yes/no question at the beginning of each set. For example ‘Are you building a bridge?’, if the answer is ‘no’, the users isn’t ask any more questions about bridges and more onto the next set.

A user will often not just be doing something with a bridge. They might be using scaffolding too. Users going through all sets of questions, allows this to be built into the routing. The value from each set is stored in the browser. These then tell the result page what to show: ‘you don’t need permission’, ‘register details’, ‘apply for a permit.’
